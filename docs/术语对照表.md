## 翻译原则

### 1. 保留英文的情况

**必须保留：**
- ✅ 专有名词：GitHub, Git, npm, Node.js, TypeScript
- ✅ 缩写词：API, CLI, JSON, YAML, HTML, CSS
- ✅ 技术术语：Promise, async/await, React, Vue
- ✅ 命令名称：`/learn`, `/plan`, `/code-review`
- ✅ 文件名：`README.md`, `package.json`
- ✅ 代码中的所有英文

**可以保留：**
- ⚠️ Hooks, Skills, Agents（如果觉得翻译不自然）
- ⚠️ MCP, TDD（业界通用的缩写）
- ⚠️ Pattern, Context（如果上下文清晰）

### 2. 翻译技巧

**代码相关：**
```markdown
❌ 不要翻译代码
✅ 保持代码原样

❌ 不要翻译变量名
✅ const marketSearchQuery = 'election'

❌ 不要翻译函数名
✅ async function fetchMarketData()
```

**命令相关：**
```markdown
❌ 不要翻译命令本身
✅ 使用 /learn 命令提取模式

✅ 可以翻译描述
✅ /learn - 从会话中提取可重用模式
```

**技术术语：**
```markdown
✅ 使用通用中文翻译
Code Review → 代码审查
Unit Test → 单元测试
Error Handling → 错误处理

⚠️ 保留业界常用的英文
TDD → TDD（或测试驱动开发）
CI/CD → CI/CD（或持续集成/持续部署）
```

### 3. 格式规范

**Markdown 格式：**
- ✅ 保持原有的标题层级
- ✅ 保持代码块格式
- ✅ 保持列表格式
- ✅ 保持链接格式

**特殊标记：**
- ✅ 保留 emoji（✅ ❌ ⭐ 等）
- ✅ 保留高亮标记（**粗体**，*斜体*）
- ✅ 保留代码标记（`code`）

### 4. 一致性要求

**术语一致：**
- 同一术语在全文中保持一致
- 使用本对照表进行统一

**风格一致：**
- 使用简洁明了的中文表达
- 避免翻译腔（"的"过多、语序生硬）
- 符合中文技术文档习惯

**格式一致：**
- 标题层级统一
- 列表格式统一
- 代码块格式统一

## 常见句式翻译

### 1. 祈使句

```
✅ 使用 /learn 命令
❌ 请使用 /learn 命令（避免多余的"请"）

✅ 创建新的技能文件
❌ 你应该创建新的技能文件（避免"你应该"）
```

### 2. 说明句

```
✅ 这个命令会分析会话并提取模式
❌ 此命令将会去分析会话并提取出模式（避免"此"、"将会"、"出"）

✅ Hook 在每次文件编辑后自动运行
❌ Hook 会在每次文件编辑之后自动地运行（避免"地"、"之后"）
```

### 3. 条件句

```
✅如果文件存在，则格式化它
❌ 若文件存在的话，那么就格式化它（避免"若"、"的话"、"那么就"）

✅ 当检测到 console.log 时，发出警告
❌ 当检测到了 console.log 的时候，发出警告（避免"到了"、"的时候"）
```

## 特殊情况处理

### 1. 代码注释

```javascript
// ✅ 可以翻译注释
// ❌ BAD: Unclear names
// ✅ 不好：不清晰的命名

// ✅ 保持代码不变
const marketSearchQuery = 'election'  // 不翻译变量名和代码
```

### 2. 配置文件

```json
// ✅ 翻译 description 字段
{
  "name": "planner",
  "description": "专家规划专员，专注于创建可执行的实施计划"
}

// ❌ 不要翻译键名
```

### 3. 命令行

```bash

/learn

# ❌ 不要翻译命令本身
/学习  # 错误！
```
