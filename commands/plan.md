# Plan 命令

此命令调用 **planner**（规划器）代理在编写任何代码之前创建全面的实施计划。

## 此命令的作用

1. **重述需求** - 明确需要构建的内容
2. **识别风险** - 提出潜在问题和阻塞因素
3. **创建步骤计划** - 将实施分解为各个阶段
4. **等待确认** - 在继续之前必须获得用户批准

## 何时使用

在以下情况使用 `/plan`：
- 开始新功能
- 进行重大架构变更
- 处理复杂的重构
- 多个文件/组件将受到影响
- 需求不明确或模糊

## 工作原理

规划器代理将：

1. **分析请求** 并用清晰的术语重述需求
2. **分解为阶段** 包含具体、可执行的步骤
3. **识别依赖** 组件之间的关系
4. **评估风险** 和潜在的阻塞因素
5. **估算复杂度** (高/中/低)
6. **展示计划** 并等待你的明确确认

## 使用示例

```
User: /plan I need to add real-time notifications when markets resolve

Agent (planner):
# 实施计划：实时市场结算通知

## 需求重述
- 当用户关注的的市场结算时向用户发送通知
- 支持多种通知渠道（应用内、邮件、webhook）
- 确保通知可靠送达
- 包含市场结果和用户持仓结果

## 实施阶段

### 阶段 1：数据库架构
- 添加 notifications 表，包含列：id、user_id、market_id、type、status、created_at
- 添加 user_notification_preferences 表用于渠道偏好设置
- 在 user_id 和 market_id 上创建索引以提升性能

### 阶段 2：通知服务
- 在 lib/notifications.ts 中创建通知服务
- 使用 BullMQ/Redis 实现通知队列
- 为失败传递添加重试逻辑
- 创建通知模板

### 阶段 3：集成点
- 钩入市场结算逻辑（当状态变为"resolved"时）
- 查询市场中所有有持仓的用户
- 为每个用户排队通知

### 阶段 4：前端组件
- 在 header 中创建 NotificationBell 组件
- 添加 NotificationList 模态框
- 通过 Supabase 订阅实现实时更新
- 添加通知偏好设置页面

## 依赖项
- Redis（用于队列）
- 邮件服务（SendGrid/Resend）
- Supabase 实时订阅

## 风险
- 高：邮件送达率（需要 SPF/DKIM）
- 中：每个市场 1000+ 用户时的性能
- 中：市场频繁结算时的通知垃圾问题
- 低：实时订阅开销

## 估算复杂度：中
- 后端：4-6 小时
- 前端：3-4 小时
- 测试：2-3 小时
- 总计：9-13 小时

**等待确认**：是否继续执行此计划？(yes/no/modify)
```

## 重要说明

**关键**：规划器代理在你用 "yes"、"proceed" 或类似肯定回应明确确认计划之前，**不会**编写任何代码。

如果想要更改，请回复：
- "modify: [你的更改]"
- "different approach: [替代方案]"
- "跳过阶段 2，先做阶段 3"

## 与其他命令的集成

规划之后：
- 使用 `/tdd` 进行测试驱动开发实施
- 如果出现构建错误，使用 `/build-and-fix`
- 使用 `/code-review` 审查完成的实施
